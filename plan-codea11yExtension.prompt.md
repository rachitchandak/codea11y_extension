## Plan: Extension Auth + Server Migration

Move the extension off its embedded localhost-only server contract and onto authenticated endpoints hosted by server_to_be_used. To reduce breakage, keep the current audit engine behavior intact behind a compatibility layer in server_to_be_used first, then merge the audit/report services into server_to_be_used without changing the extension contract a second time. Authentication should be embedded in the sidebar UI, but the extension host should own token storage and server calls so credentials are not left in webview state.

**Steps**
1. Phase 1: Freeze the current extension/server contract. Enumerate the exact request and response shapes used by the extension today: /agent/start NDJSON streaming, /reports/open, /reports/:id, /reports/project-snapshot, and /ignore-issue. This phase is already largely known from src/extension.ts runAgentAudit and src/serverClient.ts. The migration target should preserve these semantics behind a new authenticated base URL. This phase blocks all later backend and client work.
2. Phase 2: Define the new public API surface in server_to_be_used. Add extension-facing routes under a stable namespace such as /api/extension or /api/audit and make them the only endpoints the extension will call. Keep request/response payloads aligned with the current contract so extension-side changes are mostly transport and auth related. Decide once whether /agent/start remains NDJSON or moves to SSE; recommended is to keep NDJSON for the first migration because src/extension.ts already parses it line-by-line. Depends on 1.
3. Phase 3: Add an authenticated compatibility bridge in server_to_be_used. Create route handlers that require authenticate and requireApproved, then forward calls to the legacy audit engine for all report and agent endpoints, including streaming pass-through for /agent/start. This satisfies the requested proxy work and provides a safe cutover path while the full merge is in progress. It also lets the extension switch to the new base URL before the legacy audit code is transplanted. Depends on 2.
4. Phase 4: Add extension authentication state and secure token storage. Introduce extension-side auth service logic to log in against server_to_be_used /api/auth/login, persist the JWT in VS Code SecretStorage, restore it on activation, validate it via /api/user/me or a dedicated auth status endpoint, and attach Authorization headers on every server request. The extension should stop spawning or depending on anonymous localhost access as the default path. Depends on 2 and can begin in parallel with 3 once the auth contract is fixed.
5. Phase 5: Gate the sidebar and report workflows until authenticated. Extend the shared message contract with auth messages such as GET_AUTH_STATE, LOGIN_REQUEST, LOGIN_SUCCESS, LOGIN_FAILURE, LOGOUT, and AUTH_STATE. Update SidebarProvider to answer those messages and to reject SEND_QUERY, CREATE_CHAT, OPEN_CHAT, and related actions when there is no valid authenticated session. Update the React sidebar to render a login screen first and only reveal ChatListView and ChatView after auth succeeds. Depends on 4.
6. Phase 6: Build the embedded login experience in the sidebar webview. Reuse the visual content and validation rules from server_to_be_used/public/vscode-login.html and server_to_be_used/public/login.html, but do not rely on browser redirect callbacks for the primary path. Recommended implementation: render the login form in src/webview/sidebar, post credentials to the extension host, let the extension host call /api/auth/login, then send only auth state and profile details back to the webview. This preserves the “login page in chat sidebar UI” requirement while keeping the token out of DOM-local storage. Depends on 4 and 5.
7. Phase 7: Refactor extension transport into a single authenticated client. Replace hardcoded http://localhost:7544 usage in src/extension.ts and src/serverClient.ts with a configurable base URL for server_to_be_used, plus a shared request helper that injects Authorization and handles 401/403 consistently. Centralize health checks, login status checks, and report/audit fetches in one client module so the sidebar, report panel, and activation lifecycle all behave consistently. Depends on 4.
8. Phase 8: Cut the extension over to the new backend without merging audit internals yet. Update activation so it no longer unconditionally spawns the old local server. If a temporary bridge still needs the legacy audit engine running locally, make that an explicit compatibility mode behind config or a backend-side deployment detail rather than the extension’s default assumption. The extension should now talk only to server_to_be_used. Depends on 3, 4, 5, 6, and 7.
9. Phase 9: Merge the audit/report engine into server_to_be_used. Move or adapt the legacy server modules so server_to_be_used owns MainAgent, ReportService, ToolWrapper, LLMClient integration for audit flows, ContrastProcessor, issue helpers, source inventory, prompts, and the report persistence schema. Preserve behavior first: file hashing, caching, deduplication, validation pass, and streaming events must remain equivalent. Keep the extension-facing route shapes created in Phase 2 unchanged. Depends on 3 and 8.
10. Phase 10: Consolidate persistence and configuration. Decide whether to keep audit/report tables in the same database used by server_to_be_used or to isolate them behind a separate DB module within the same service. Recommended: keep one service but two logical database modules initially, then converge only if the schema overlap becomes useful. Also decide how legacy API.txt settings map to the per-user configuration model in server_to_be_used; likely the audit engine should consume the authenticated user’s API config rather than shared file-based secrets. Depends on 9.
11. Phase 11: Remove the proxy bridge and legacy startup path. Once the merged handlers in server_to_be_used produce equivalent results, swap the compatibility bridge routes to direct in-process handlers or preserve the same route names and remove the forwarder behind them. Then delete extension-side local server startup logic and any now-unused compatibility configuration. Depends on 9 and 10.
12. Phase 12: Verify end-to-end behavior with authentication, approval gating, and audit/report flows. Validate login failure, unapproved user handling, restored sessions, expired tokens, report retrieval, project snapshots, ignore-issue sync, and streamed audit progress. This phase depends on all prior phases and should be repeated after the proxy cutover and again after the full merge.

2. Verify that an unauthenticated user opening the sidebar sees only the login screen and cannot send chat queries, open project audits, or retrieve reports.

5. Verify that /agent/start streaming still updates the sidebar activity cards and todo state without parser changes after routing through server_to_be_used.
6. Verify report retrieval parity for single-file open, project snapshot, and ignore-issue behavior before and after the audit engine is fully merged.

8. Run existing build/test commands for both the extension and server_to_be_used where available, plus manual end-to-end login and audit flows.

**Decisions**
- Included: embedded sidebar login as the primary UX, extension-wide auth gating, proxy bridge in server_to_be_used, and a full eventual merge of legacy audit logic into server_to_be_used.
- Recommended: use extension-host-managed auth and SecretStorage rather than storing JWTs in the webview or relying on the current browser redirect callback as the primary path.
- Recommended: preserve NDJSON for agent streaming during migration to avoid changing the extension stream parser and server internals simultaneously.
- Excluded from the first cut: redesigning the audit UX, changing LLM prompt behavior, or reworking report semantics beyond what is required for auth and backend migration.

**Further Considerations**
1. Authentication transport: best near-term security is sidebar form in webview plus extension-host fetch and SecretStorage. A pure webview fetch model is faster to build but weaker because the token lives in the DOM runtime.
2. Backend merge strategy: even though the target is a full merge, keeping the proxy bridge first avoids a double client migration and gives a measurable parity checkpoint before moving MainAgent and ReportService.
3. Database strategy: merging auth and audit into one service does not require immediate table unification. Keeping separate modules against one or two SQLite files initially lowers schema risk while the audit engine is being ported.
